Estructura del Proyecto

Arquitectura de la Aplicación

La aplicación sigue una arquitectura en capas con separación de responsabilidades:

BibliotecaNormal/
├── Models/           # Entidades y modelos de datos
├── Controllers/     # Lógica de negocio y controladores
├── Views/             # Interfaces de usuario
├── Data/              # Acceso a datos y repositorios
├── Services/         # Servicios de aplicación
└── Utilities/        # Utilidades y helpers

Organización de Carpetas

El proyecto está organizado siguiendo las mejores prácticas de ASP.NET:

Solution BibliotecaNormal
├── BibliotecaNormal.Web  # Proyecto principal (ASP.NET)
│  ├── App_Data
│  ├── Content          # Hojas de estilo, imágenes
│  ├── Controllers      # Controladores MVC
│  ├── Models            # Modelos de datos
│  ├── Views             # Vistas Razor
│  ├── Scripts           # Archivos JavaScript
│  └── Global.asax     # Configuración global
├── BibliotecaNormal.Data  # Capa de acceso a datos
└── BibliotecaNormal.Models # Modelos compartidos

Operaciones CRUD

Implementación de CRUD en ASP.NET

Las operaciones CRUD (Create, Read, Update, Delete) se implementaron utilizando Entity Framework como ORM y el patrón Repository para abstraer el acceso a datos.

Estructura típica de un controlador:

public class LibrosController : Controller
{
    private readonly IRepository<Libro> _repository;

    public LibrosController(IRepository<Libro> repository)
    {
        _repository = repository;
    }

    // GET: Libros
    public ActionResult Index()
    {
        return View(_repository.GetAll());
    }

    // GET: Libros/Create
    public ActionResult Create()
    {
        return View();
    }

    // POST: Libros/Create
    [HttpPost]
    public ActionResult Create(Libro libro)
    {
        _repository.Add(libro);
        _repository.Save();
        return RedirectToAction("Index");
    }

    // Implementaciones similares para Edit, Details, Delete
}

Control de Versiones con Git

Flujo de trabajo con Git

El proyecto utiliza Git para el control de versiones, siguiendo el flujo de trabajo de ramas feature-based:

# Inicializar repositorio
git init
git add .
git commit -m "Initial commit"

# Crear una rama para una nueva característica
git checkout -b feature/crud-libros

# Trabajar en la característica y hacer commits
git add .
git commit -m "Implementar operaciones CRUD para libros"

# Fusionar con la rama principal
git checkout main
git merge feature/crud-libros

# Subir cambios al repositorio remoto
git push origin main

Estructura de ramas

Se utiliza la siguiente estructura de ramas:

main             # Rama principal de producción
develop        # Rama de integración
feature/*    # Ramas para nuevas características
hotfix/*     # Ramas para correcciones urgentes
release/*    # Ramas para preparar releases
